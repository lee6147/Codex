## 1. Claude Code

### 강점

- 2026-05-01 <code>v2.1.126</code>은 gateway model picker, pasted auth code 로그인, project purge, Windows PowerShell 7 탐지처럼 실제 장기 CLI 세션에서 자주 막히던 지점을 겨냥했다.
- 셸 중심 작업과 반복 수정 루프에서 안정적으로 오래 일하게 만드는 운영 품질이 여전히 강점이다.
- 공식 릴리스와 최근 기사 모두 Claude Code가 “멋진 데모”보다 실제 개발 루프의 마찰을 줄이는 방향으로 가고 있음을 보여 준다.

### 약점

- 외부 기사에서 가장 크게 보이는 주제는 여전히 권한 남용과 안전장치 부족이다. PocketOS 사고는 Claude 기반 에이전트 생태계 전반이 안고 있는 위험을 상기시킨다.
- GUI 중심 협업이나 시각적 팀 오케스트레이션은 Cursor보다 덜 직접적이다.
- 운영 품질은 강하지만, 이를 안전하게 쓰기 위한 승인 규칙과 환경 분리가 약하면 장점이 곧 리스크가 된다.

### 추천 사용 시나리오

- 한 세션 안에서 분석, 수정, 검증을 오래 이어 가는 CLI 중심 개발 작업
- 로컬 스크립트, 셸 명령, 리포지토리 탐색을 끊기지 않게 붙여 쓰는 자동화 흐름
- GUI보다 실행 일관성과 운영 복구성을 더 중시하는 사용자

### 피해야 할 사용 시나리오

- 승인 단계와 권한 경계를 설계하지 않은 채 바로 고권한 자동화를 크게 여는 환경
- 팀 운영면보다 IDE 안의 시각적 관리가 더 중요한 조직

### 최근 업데이트 반영 관찰

- Claude Code의 현재 메시지는 “더 화려함”보다 “더 안 끊김”이다.
- 다음 비교 포인트는 기능 수보다 인증 복구력, Windows 셸 친화성, 긴 세션 안정성, 승인 UX가 된다.

### 현재 시점 총평

Claude Code는 여전히 가장 실전적인 CLI형 도구다. 이번 창에서도 성능 과시보다 장기 작업을 덜 끊기게 만드는 운영 품질이 강하게 보였다.

## 2. Cursor

### 강점

- 2026-05-01 Team Marketplace 업데이트는 저장소 연결 없이도 팀 플러그인 배포 정책을 만들 수 있게 하며, Cursor를 팀 공통 작업 계약 배포 플랫폼으로 한 단계 더 밀어 올렸다.
- 외부 기사들은 Cursor를 AI-native editor가 아니라 agent workspace로 읽는다. 즉 단일 보조 기능보다 병렬 agent 운영과 코드베이스 인식이 핵심 경쟁력이다.
- 제품팀이 AI 프로토타입을 만들고 엔지니어가 프로덕션으로 끌어올리는 새 협업 흐름에서 Cursor는 가장 공격적인 위치를 차지하고 있다.

### 약점

- 역할 경계가 흐려질수록 기대치 충돌이 커진다. 제품팀의 프로토타입과 엔지니어링의 운영 기준을 분리하지 않으면 생산성이 오히려 어수선해질 수 있다.
- 보안 감사, 승인 UX, 로그, 복구 절차가 약하면 강한 자동화 실행면이 위험해진다.
- CLI 순수주의자나 로컬 단순성을 중시하는 팀에게는 일부 장점이 덜 직접적으로 느껴질 수 있다.

### 추천 사용 시나리오

- IDE 안에서 병렬 작업과 작업 분리를 시각적으로 관리하려는 팀
- 팀 규칙 배포와 plugin/skill 유통까지 같이 다루려는 조직
- 제품팀과 엔지니어링 팀이 프로토타입과 구현 사이의 핸드오프를 빠르게 돌리고 싶은 환경

### 피해야 할 사용 시나리오

- 팀 기준 없이 각자 에이전트를 임의로 굴리는 환경
- 최소한의 승인 체계만 가진 상태에서 고권한 agent 실행을 빠르게 여는 경우

### 최근 업데이트 반영 관찰

- Cursor는 이번 창에서 “좋은 에디터”보다 “운영 가능한 팀 플랫폼”이라는 메시지를 더 강하게 냈다.
- 외부 기사도 같은 방향을 확인한다. 성능 비교보다 역할 설계와 조직 적용이 더 중요한 주제가 됐다.

### 현재 시점 총평

Cursor는 이번 창에서 가장 선명한 팀 운영 모멘텀을 보여 줬다. 다만 그 힘은 규칙 배포와 기대치 조율을 감당할 준비가 된 팀에게 더 잘 맞는다.

## 3. OpenAI Codex

### 강점

- 플랜 문서와 rate card가 최근 갱신되며 접근 범위, 데이터 제어 기능, 토큰 기반 비용 구조를 더 명확하게 설명한다.
- 외부 기사와 공식 문서를 함께 보면 Codex는 코드 생성기보다 control plane에 가까운 제품으로 읽힌다.
- Memories, Automations, in-app browser, Computer Use 같은 기능 묶음은 반복 업무와 장기 위임 관점에서 구조적으로 강하다.

### 약점

- 최근 읽을 만한 외부 실사용 기사보다 공식 정책 문서가 더 많이 잡힌다. 즉 도구 이해에 문서 해석 부담이 크다.
- Help Center는 절대 수정 시각을 숨기는 경우가 있어 최신성 추적이 불편하다.
- 비용 구조가 토큰 기반으로 더 직접 연결되면서, 사용량 관리까지 운영자가 챙겨야 한다.

### 추천 사용 시나리오

- 반복 리서치, 문서 작업, 코드 작업을 하나의 에이전트 운영면으로 묶고 싶은 조직
- 권한 제어, 반복 자동화, 장기 메모리와 workflow continuity가 중요한 환경
- 코드 외에도 브라우저와 컴퓨터 사용까지 포함한 다중 실행면 자동화를 원하는 팀

### 피해야 할 사용 시나리오

- 즉시성 높은 IDE 보조만 원하는 개인 사용자
- 운영 문서와 비용 구조를 읽고 관리할 여력이 없는 작은 실험성 사용

### 최근 업데이트 반영 관찰

- 이번 창의 Codex 핵심은 “무엇이 새로 추가됐나”보다 “누가 어떤 플랜으로 어떻게 쓰며 무엇을 저장하나”였다.
- 즉 경쟁력은 모델 데모보다 control plane 정합성, 가격 투명성, 인간 검토 전제를 어떻게 설명하는가에 있다.

### 현재 시점 총평

Codex는 가장 하네스 지향적이고 조직형 운영에 맞는 도구다. 이번 창에서도 화려함은 덜했지만 실제 도입 관점에서는 정책과 운영 구조가 더 선명해졌다.
