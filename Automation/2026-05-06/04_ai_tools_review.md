## 1. Claude Code

### 강점

- 최신 공식 릴리스 `v2.1.128`은 worktree, plugin zip, parallel shell, sub-agent progress cache 같은 장기 세션 운영 품질을 직접 보강했다.
- CLI 안에서 읽기, 수정, 검증, 하위 실행을 오래 이어 가는 감각이 여전히 가장 강하다.
- Auto Mode와 Claude Security 흐름을 보면 단순 코딩 보조가 아니라 승인 경계를 가진 실행 런타임, 그리고 보안 운영 도구로까지 확장되고 있다.

### 약점

- 권한 구조와 승인 정책을 설계하지 않으면 Auto Mode의 장점이 바로 리스크로 바뀐다.
- 팀용 관리면은 강해지고 있지만, 여전히 Cursor보다 시각적 제품화는 덜 직접적이다.
- 사용량과 운영 비용을 별도 관리하지 않으면 성능 체감이 좋아질수록 비용 부담도 같이 커진다.

### 추천 사용 시나리오

- 큰 리포에서 터미널 기반으로 길게 이어지는 분석, 수정, 검증 루프
- 셸 명령과 코드 편집을 끊지 않고 묶어 쓰는 개발자
- 하위 에이전트, worktree, 검증 중심으로 실행 품질을 끌어올리고 싶은 팀

### 피해야 할 사용 시나리오

- 권한 경계와 검증 루프 없이 곧바로 고권한 자동화를 열어 두는 환경
- GUI 중심 협업이 더 중요한 팀

### 최근 업데이트 반영 관점

- 이번 주 Claude Code의 핵심은 새 기능 숫자보다 “오래 돌릴 때 덜 깨지고 덜 막히는가”다.
- 즉 경쟁력의 중심이 모델 성능보다 실행 루프의 복구성과 권한 체계로 이동하고 있다.

### 현재 시점 총평

Claude Code는 여전히 가장 실전적인 CLI형 에이전트다. 다만 지금의 진짜 강점은 똑똑함만이 아니라, 복잡한 세션을 계속 이어 가는 운영 감각에 있다.

## 2. Cursor

### 강점

- 2026-05-04 공식 업데이트는 모델 허용/차단, soft spend limit, usage analytics를 surface 단위로 다루게 하며 팀 운영 도구 성격을 더 선명하게 만들었다.
- SDK와 Security Review 흐름이 붙으면서 에디터 안 에이전트가 제품팀과 플랫폼팀의 공용 런타임으로 확장되고 있다.
- GUI 안에서 병렬 agent, review, security, cloud runtime을 시각적으로 관리하기 쉬운 점이 강하다.

### 약점

- 로컬 단순성과 CLI 순수성을 중시하는 사용자에게는 무게중심이 다소 과하다.
- 기능이 많아질수록 규칙 없이 각자 쓰면 조직 내 품질 편차가 커질 수 있다.
- 제품 방향이 엔터프라이즈 운영층으로 강하게 기울어, 개인용 “가벼운 AI 에디터” 기대와는 점점 멀어진다.

### 추천 사용 시나리오

- IDE 안에서 병렬 agent와 시각적 상태 관리를 함께 원하는 팀
- 보안 리뷰, usage 통제, plugin 정책까지 조직 차원에서 배포하려는 환경
- SDK를 이용해 에이전트를 CI/CD나 내부 서비스에 재사용하려는 팀

### 피해야 할 사용 시나리오

- 편집기 기능만 원하면서 운영 플랫폼 복잡도는 감당하고 싶지 않은 경우
- 팀 규칙 없이 각자 agent를 임의로 굴리는 조직

### 최근 업데이트 반영 관점

- Cursor의 현재 메시지는 “좋은 AI IDE”보다 “팀용 agent operating surface”다.
- 이번 주는 특히 모델 선택권과 비용 가시성을 묶어 통제 가능한 플랫폼으로 진화하는 방향이 뚜렷했다.

### 현재 시점 총평

Cursor는 이번 주에도 팀 운영과 제품화 방향이 가장 선명했다. 가장 강력한 GUI 계열 선택지지만, 그 강점은 규칙과 검토 체계가 있는 조직일수록 크게 작동한다.

## 3. OpenAI Codex

### 강점

- 최근 갱신된 플랜 문서는 CLI, IDE extension, app, web, plugins, RBAC, Compliance API, Local/Cloud 경계를 한 번에 정리한다.
- `/goal`과 같은 장기 실행 기능을 중심으로 “잠깐 쓰는 도구”보다 “오래 맡겨 두는 런타임” 쪽 강점이 분명하다.
- app, automations, in-app browser, Computer Use까지 합쳐 가장 넓은 운영면을 가진 도구라는 점이 구조적 강점이다.

### 약점

- 문서와 정책이 자주 바뀌고 Help Center는 정확한 편집 시각을 숨기는 경우가 있어, 실제 운영자는 계속 최신 상태를 확인해야 한다.
- 넓은 실행면은 곧 더 많은 정책 결정과 권한 설계를 요구한다.
- 즉시성 높은 IDE 보조만 원하는 사용자에게는 무게가 다소 크다.

### 추천 사용 시나리오

- 로컬 실행과 cloud delegation을 함께 쓰는 팀
- 반복 리서치, 코드 작업, 리뷰, 자동화를 하나의 control plane으로 묶고 싶은 조직
- spec 기반 장기 실행과 운영 계약을 중시하는 환경

### 피해야 할 사용 시나리오

- 운영 문서와 비용 구조를 읽고 관리할 시간적 여력이 없는 가벼운 실험
- 즉답형 에디터 보조만 원하는 개인 사용자

### 최근 업데이트 반영 관점

- Codex의 최근 강점은 기능 쇼케이스보다 정책과 실행면 정합성에 있다.
- 이번 주 기사와 공식 문서는 모두 “좋은 spec, 좋은 경계, 좋은 장기 상태 관리”가 성능만큼 중요하다고 말한다.

### 현재 시점 총평

Codex는 가장 넓은 운영면을 가진 도구다. 그만큼 강력하지만, 제대로 쓰려면 기능보다 먼저 정책과 실행 계약을 읽어야 한다.

## 이번 주 총평

이번 주 시장 신호는 명확했다. 승부처는 모델 데모가 아니라 운영면이다. Claude Code는 복구성과 CLI 실행 감각, Cursor는 통제 가능한 팀 플랫폼화, Codex는 가장 넓은 control plane에서 강점을 보였다. 반대로 세 도구 모두 권한 경계, 검토 책임, 비용 추적을 무시하면 장점이 곧 부담이 된다.
