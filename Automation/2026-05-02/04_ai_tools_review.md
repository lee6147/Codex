## 1. Claude Code

### 강점

- `v2.1.126`은 gateway model picker, pasted auth code 로그인, 프로젝트 purge UI, Windows PowerShell 7 탐지, OAuth refresh token 개선처럼 실제 사용 흐름을 덜 끊기게 만드는 요소에 집중했다.
- `v2.1.123`과 이어 보면 Claude Code는 여전히 릴리스 빈도가 높고, 작은 장애를 빠르게 다루는 CLI형 실전 도구의 성격이 강하다.
- 긴 세션, 반복 수정, 셸 중심 작업에서 체감 품질을 높이는 방향이 명확하다.

### 약점

- 이번 창의 변화는 화려한 신기능보다 운영 패치라서 외부 체감 모멘텀은 Cursor보다 약하게 보일 수 있다.
- 기사 창에서는 프롬프트 주입과 권한 오남용 우려가 반복 노출됐다. 셸에 가까운 도구일수록 승인 UX와 검토 흐름의 완성도가 중요하다.
- 조직 도입 관점에서는 모델 성능보다 토큰, 권한, 저장소 쓰기 범위를 먼저 설계해야 한다.

### 추천 사용 시나리오

- 한 세션에서 분석, 수정, 검증을 오래 이어 가는 CLI 중심 개발 작업
- 로컬 스크립트, 훅, 셸 명령, 리포지토리 탐색을 끊기지 않게 붙여 쓰는 자동화 흐름
- GUI보다 실행 일관성과 운영 복구성을 우선하는 사용자

### 피해야 할 사용 시나리오

- 승인 단계와 권한 경계를 따로 설계하지 않고 바로 셸 자동화를 크게 여는 환경
- 설정 최소화와 시각적 관리가 더 중요한 팀

### 최근 업데이트 반영 관찰

- Claude Code의 현재 메시지는 “더 화려함”보다 “더 안 끊김”이다.
- 따라서 다음 비교 포인트는 기능 수보다 안정성, 권한 검토 흐름, 운영층 품질이 되어야 한다.

### 현재 시점 총평

Claude Code는 여전히 가장 실전적인 CLI형 도구다. 이번 창에서는 대형 제품 발표보다 운영 복구성과 세션 안정성이 더 중요하다는 점을 다시 보여줬다.

## 2. Cursor

### 강점

- 2026-04-29 SDK, 2026-04-30 Security Review, 2026-05-01 Team Marketplace의 조합은 이번 창에서 가장 완성도 높은 제품 서사를 만들었다.
- UI 기반 병렬 작업, agent 호출 SDK, 사후 보안 검토, 팀 배포 마켓까지 한 제품 안에 묶으려는 방향이 분명하다.
- IDE 친화성과 조직 확장성을 동시에 가져가려는 전략이 가장 공격적이다.

### 약점

- 실행면이 넓어질수록 승인 UX, 보안 검토, 정책 통제가 따라오지 않으면 사고 비용도 빠르게 커진다.
- The Guardian 보도처럼 프롬프트 주입과 승인 설계 문제는 “좋은 사용감”만으로 덮을 수 없는 리스크다.
- 순수 CLI 중심 문화에서는 Cursor의 강점 일부가 덜 직접적으로 느껴질 수 있다.

### 추천 사용 시나리오

- IDE 안에서 병렬 작업과 작업 분리를 시각적으로 관리하려는 팀
- SDK와 팀 규칙 배포까지 연결해 Cursor를 내부 개발 플랫폼처럼 쓰려는 조직
- GUI와 에이전트 자동화를 함께 가져가고 싶은 사용자

### 피해야 할 사용 시나리오

- 최소한의 권한 체계만 갖춘 상태에서 곧바로 대규모 agent 실행을 여는 경우
- 화면보다 셸을 우선하고, 편집기 플랫폼성보다 명령줄 직접성을 더 중시하는 환경

### 최근 업데이트 반영 관찰

- Cursor는 이번 주에 “좋은 AI 에디터”를 넘어 “운영 가능한 agent 플랫폼”으로 프레이밍을 강화했다.
- SDK와 Team Marketplace가 자리 잡으면 경쟁 축은 IDE 보조가 아니라 팀 단위 작업 오케스트레이션 전체가 된다.

### 현재 시점 총평

Cursor는 이번 창에서 가장 명확한 확장 메시지를 냈다. 다만 그 확장성은 보안 검토와 승인 정책을 감당할 준비가 된 팀에게 더 잘 맞는다.

## 3. OpenAI Codex

### 강점

- 플랜 문서가 Free·Go의 제한적 포함, Plus·Pro·Business·Enterprise/Edu 조건, memories·automations·browser·computer use를 한 번에 묶어 설명하면서 제품 구조를 더 읽기 쉽게 만들었다.
- Rate card와 Security 문서까지 이어 보면 Codex는 orchestration control plane으로서의 운영 기준을 빠르게 문서화하고 있다.
- 조직 도입 시 비용, 권한, 보안 경계를 같이 설계하기 좋은 형태로 정리되고 있다.

### 약점

- 이번 창의 변화가 대부분 문서 갱신이라, 외부에서 느끼는 “제품 모멘텀”은 Cursor보다 약하다.
- Help Center가 절대 수정일을 숨겨 최신성 확인이 한 단계 불편하다.
- 초보 사용자는 기능 이해보다 플랜, 과금, 보안 구조를 먼저 읽어야 해서 진입 장벽이 남는다.

### 추천 사용 시나리오

- 자동화, 반복 리서치, 역할 분리, 권한 통제가 중요한 조직형 작업
- 코드 외에도 문서, 브라우저 작업, 컴퓨터 사용까지 하나의 실행 하네스로 묶으려는 팀
- 비용과 권한을 명시적으로 관리할 수 있는 환경

### 피해야 할 사용 시나리오

- 즉시성 높은 IDE 보조만 원하는 개인 사용자
- 운영 문서와 보안 경계를 읽고 관리할 여력이 없는 작은 실험성 사용

### 최근 업데이트 반영 관찰

- 이번 창의 Codex 핵심은 “무엇을 더 해 주나”보다 “어떤 플랜과 비용, 보안 경계 안에서 굴리나”였다.
- 즉 제품 데모보다 운영 문서의 품질이 더 중요해진 시점이다.

### 현재 시점 총평

Codex는 가장 하네스 지향적이고 조직형 운영에 맞는 도구다. 이번 창의 문서 중심 흐름은 화려함은 약하지만, 실제 도입 관점에서는 오히려 핵심적인 진전이다.
