## 1. Claude Code

### 강점

- 2026-04-16 기준 Opus 4.7 xhigh, Auto mode 확대, `/less-permission-prompts`, `/ultrareview` 까지 들어오며 CLI 에이전트 운영 밀도가 더 높아졌다.
- 같은 날 v2.1.112로 Auto mode 불안정성을 바로 잡은 점은 기능보다 운영 안정성을 우선하는 태도로 읽힌다.
- 권한, 세션 복귀, 클라우드 리뷰, 데스크톱/웹/원격 제어가 하나의 워크플로로 이어져 하네스 중심 팀과 궁합이 좋다.

### 약점

- 기능과 운영 문서가 changelog, docs, whats-new, 도움말로 나뉘어 있어 추적 비용이 높다.
- 텍스트와 명령 중심 경험이 강해 GUI 친화적 협업 환경에는 여전히 진입장벽이 있다.
- 빠른 배포 주기 때문에 팀 규칙 파일과 운영 문서를 자주 손봐야 한다.

### 추천 사용 시나리오

- 터미널 중심 개발팀이 장기 작업, 리뷰 자동화, 세션 복귀, 권한 통제를 함께 굴릴 때.
- 규칙 파일, 메모리, 승인 경계, 검증 루프를 직접 설계하는 엔지니어링 조직.
- 한 번의 답변보다 반복 실행과 운영 통제가 더 중요한 백엔드·플랫폼 팀.

### 피해야 할 사용 시나리오

- IDE 안에서 시각적으로 여러 에이전트 결과를 동시에 비교해야 하는 협업 중심 팀.
- 명령어와 설정 파일보다 즉시 눈에 보이는 그래픽 워크플로를 우선하는 사용자.

### 최근 업데이트 반영 관점

- 최신 변화의 본질은 “더 똑똑한 모델”보다 “덜 끊기고 덜 묻는 운영 루틴”이다.
- Auto mode 안정화와 `/less-permission-prompts` 는 실사용 마찰을 줄이는 쪽에 직접 연결된다.

### 현재 시점 총평

Claude Code는 여전히 가장 강한 CLI 운영형 코딩 에이전트다. 최신 변화도 모델 자랑보다 승인 마찰 축소, 세션 유지, 장기 리뷰, 반복 실행 같은 운영성 강화에 집중돼 있다.

## 2. Cursor

### 강점

- Canvases로 에이전트 응답을 다이어그램, 차트, 표, diff가 있는 검토 가능한 결과물로 바꾸며 설명력과 공유성이 크게 올라갔다.
- Cursor 3 이후 제품 철학이 파일 편집기보다 병렬 에이전트 워크스페이스에 맞춰졌고, 그 방향이 2026-04-15 업데이트까지 일관되게 이어진다.
- 로컬-클라우드 핸드오프, 타일 뷰, 음성 입력 개선처럼 여러 에이전트를 동시에 조정하는 사용자 경험이 강하다.

### 약점

- 제품 표면이 빠르게 변해 익숙해질 즈음 다시 학습 비용이 든다.
- IDE 정체성을 기대한 사용자는 에이전트 워크스페이스 전환을 오히려 부담으로 느낄 수 있다.
- 세밀한 규칙 제어는 Claude Code 같은 CLI 하네스보다 덜 직접적일 수 있다.

### 추천 사용 시나리오

- 여러 에이전트 결과를 눈으로 비교하고 검토해야 하는 제품·프론트엔드 팀.
- 시각 산출물과 디자인 검토, 로컬 미리보기, 멀티에이전트 비교가 중요한 조직.
- IDE 친화성을 유지하면서도 병렬 에이전트 운영으로 옮겨가려는 팀.

### 피해야 할 사용 시나리오

- 모든 것을 텍스트 기반 계약서와 CLI 규칙 파일로만 통제하고 싶은 팀.
- 제품 표면 변화가 잦은 도구보다 안정된 루틴을 선호하는 환경.

### 최근 업데이트 반영 관점

- 이번 주 Cursor의 핵심은 더 많은 기능 수가 아니라, 에이전트 출력을 사람이 읽고 검토하는 표면을 어떻게 재설계했는가에 있다.
- Canvases는 단순 예쁜 출력이 아니라 에이전트 결과의 전달 방식 자체를 바꾸는 변화다.

### 현재 시점 총평

Cursor는 지금 가장 제품화된 멀티에이전트 워크스페이스에 가깝다. 최근 신호도 모델보다 병렬 관리, 시각 산출물, 워크스페이스 중심 UX로 모인다.

## 3. OpenAI Codex

### 강점

- 2026-04-16 업데이트로 background computer use, 인앱 브라우저, 이미지 생성, 플러그인, 메모리, 자동화 재개, 리뷰 코멘트 처리까지 한 번에 묶으며 제품 폭을 크게 넓혔다.
- Help Center 문서와 rate card 갱신까지 이어져 기능뿐 아니라 접근 경로, 권한 통제, 과금 표면도 같이 정리되고 있다.
- 앱, CLI, IDE extension, 웹을 계정 중심으로 연결하는 서사가 분명해져 조직 도입 설명이 쉬워졌다.

### 약점

- 제품 글, 도움말, 요금 문서, 개발자 문서가 여러 표면에 흩어져 있어 최신 상태를 한 번에 파악하기 어렵다.
- 기능 범위가 매우 넓어졌기 때문에 실제 팀에 필요한 부분만 추려 도입하지 않으면 복잡도만 올라갈 수 있다.
- 세밀한 로컬 규칙 통제보다 표면 확장에 더 초점이 있어, 팀별 운영 정책은 별도 보완이 필요하다.

### 추천 사용 시나리오

- 코딩, 리뷰, 브라우저 확인, 플러그인 연동, 장기 자동화를 한 허브에서 다루고 싶은 팀.
- ChatGPT 계정 체계와 조직 통제가 이미 자리 잡은 회사.
- 개발자뿐 아니라 PM, 디자이너, 운영 인력까지 같은 에이전트 작업공간에 연결하려는 환경.

### 피해야 할 사용 시나리오

- 인터넷/앱/브라우저/자동화까지 닿는 넓은 표면보다, 좁고 엄격한 로컬 제어만 필요한 팀.
- 과금 구조와 권한 표면을 세밀히 검토하지 않고 빠르게 붙였다 떼기를 원하는 작은 실험.

### 최근 업데이트 반영 관점

- Codex의 최신 변화는 “코드를 잘 쓰는가”보다 “얼마나 넓은 개발 운영 표면을 하나로 묶는가”에 초점이 있다.
- 메모리와 자동화 재개는 반복 업무를 누적 자산으로 만드는 기능이라 장기 팀 사용에서 의미가 크다.

### 현재 시점 총평

Codex는 이번 주를 기점으로 가장 빠르게 표면을 넓히는 제품으로 다시 올라왔다. 비교 포인트도 이제 코드 작성 성능 하나가 아니라, 개발 수명주기 전체를 한 작업공간에서 얼마나 자연스럽게 잇는가로 이동했다.

## 이번 주 총평

이번 주 시장은 더 선명해졌다. 승부처는 한 번의 답변 품질보다, 에이전트를 계속 돌리고, 기억을 붙이고, 결과를 읽기 좋게 만들고, 브라우저와 앱과 PR과 문서를 하나의 흐름으로 엮는 운영 계층이다.

GitHub Trending도 같은 방향을 가리킨다. 오늘 상위권 저장소 대부분이 모델 그 자체보다 규칙 파일, 장기 메모리, 스킬 팩, 에이전트 런타임, 오픈 에이전트 템플릿을 다루고 있었다. 지금 팀이 먼저 정해야 할 것은 “어느 모델이 제일 세냐”보다 “우리 팀의 규칙, 메모리, 검증, 산출물 포맷을 어떤 운영체제로 만들 것인가”다.

## 주요 트렌드 관찰

1. 규칙 파일과 스킬 팩이 모델 성능 못지않게 중요한 차별화 포인트가 되고 있다.
2. 메모리와 장기 자동화는 실험 기능이 아니라 기본 경쟁축으로 빠르게 이동하고 있다.
3. Cursor와 Codex는 결과물과 작업 표면을 넓히고, Claude Code는 통제 가능한 실행 루틴을 더 정교하게 만들고 있다.
4. 에이전트 도구 선택은 단일 승자 찾기보다 팀 구조와 운영 문화에 맞는 도구 조합을 찾는 문제에 가까워지고 있다.
