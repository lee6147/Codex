## 1. Claude Code

### 강점

- 여전히 가장 강한 터미널 중심 실행 감각을 갖고 있다. 직전 공식 맥락인 `v2.1.126`은 PowerShell 7 탐지, PowerShell 우선화, pasted auth code 로그인, project purge처럼 장기 세션의 마찰을 줄이는 개선에 집중했다.
- CLI 안에서 읽기, 수정, 검증을 오래 이어 가는 흐름이 강점이다.
- 보안 제품군인 Claude Security로까지 확장되며 “코딩 에이전트”를 넘어 운영 도구로 넓어지고 있다.

### 약점

- 이번 공식 창에는 새 공식 항목이 없다. 즉 당장 새 방향성보다 기존 릴리스 체감 품질에 의존하는 시기다.
- 비용 추정치 상향 기사에서 보이듯, 성능 향상은 곧 운영 비용 관리 부담으로 이어진다.
- 자동화 범위를 넓힐수록 승인 단계와 복구 설계를 먼저 하지 않으면 리스크가 커진다.

### 추천 사용 시나리오

- 큰 리포에서 터미널 기반으로 길게 이어지는 분석, 수정, 검증 루프
- 셸 명령과 코드 편집을 끊지 않고 묶어 쓰는 개발자
- 실행 일관성과 현장 복구성을 더 중시하는 팀

### 피해야 할 사용 시나리오

- 권한 경계와 검증 단계를 설계하지 않은 채 바로 고권한 자동화를 여는 환경
- 비용 가시성 없이 사용량이 급증하는 팀

### 최근 업데이트 반영 관점

- 이번 주 Claude Code의 핵심은 “새 기능 대거 추가”가 아니라 “직전 운영 개선이 실제로 얼마나 안정적으로 먹히는가”다.
- 따라서 다음 비교 포인트는 기능 수보다 인증 복구력, Windows 셸 친화성, 보안 운영 연결성이다.

### 현재 시점 총평

Claude Code는 여전히 가장 실전적인 CLI형 에이전트다. 다만 지금은 새 기능보다 운영 비용과 권한 구조를 얼마나 잘 통제하느냐가 더 중요하다.

## 2. Cursor

### 강점

- Team Marketplace와 Cursor SDK 흐름이 맞물리며 개인용 AI IDE에서 팀용 agent platform으로 빠르게 이동하고 있다.
- GUI 안에서 병렬 agent, 팀 배포, plugin/marketplace 정책, cloud runtime을 다루기 쉽다.
- 제품팀과 엔지니어링 팀이 같은 agent workspace를 공유하기 좋은 방향으로 발전 중이다.

### 약점

- 이번 공식 창에도 새 changelog 항목은 없다. 방향은 분명하지만, 새 공식 신호는 잠시 멈춘 상태다.
- 기능이 강할수록 역할 경계가 흐려진다. 프로토타입과 프로덕션 기준을 분리하지 않으면 조직 내부 기대치 충돌이 커질 수 있다.
- CLI 순수주의자나 로컬 단순성을 중시하는 팀에는 일부 장점이 덜 직접적이다.

### 추천 사용 시나리오

- IDE 안에서 병렬 작업과 시각적 상태 관리를 함께 원하는 팀
- plugin, skills, MCP, review 흐름을 팀 정책으로 배포하려는 조직
- 제품팀과 엔지니어링 팀이 같은 도구 위에서 빠르게 핸드오프하려는 환경

### 피해야 할 사용 시나리오

- 규칙 없이 각자 agent를 임의로 굴리는 팀
- 편집기 기능만 원하면서 운영 플랫폼 복잡도는 감당하고 싶지 않은 경우

### 최근 업데이트 반영 관점

- Cursor의 현재 메시지는 “좋은 AI 에디터”보다 “팀용 agent operating surface”다.
- SDK와 marketplace가 함께 강화되면서, 제품의 중심이 로컬 편집기에서 조직 자동화 쪽으로 더 이동하고 있다.

### 현재 시점 총평

Cursor는 이번 주에도 팀 운영과 제품화 방향이 가장 선명했다. 다만 그 장점은 역할 정의와 검토 체계가 있는 조직일수록 더 크게 작동한다.

## 3. OpenAI Codex

### 강점

- 공식 플랜 문서가 최근 다시 갱신되며 접근 플랜, plugins, Local/Cloud 제어, Compliance API, data controls를 한 번에 정리한다.
- `0.128.0` 맥락까지 합치면 persisted `/goal`, permission profiles, plugin workflows처럼 장기 워크플로 운영면이 강하다.
- CLI, app, cloud, automations를 하나의 control plane처럼 묶어 볼 수 있다는 점이 구조적 강점이다.

### 약점

- 이번 7일 창에서도 외부 실사용 기사보다 공식 문서 비중이 높다. 즉 도구 이해에 문서 해석 비용이 크다.
- rate card와 usage 구조를 읽고 관리해야 하므로 운영자의 부담이 적지 않다.
- 도구가 넓게 열릴수록 “무엇을 할 수 있는가”보다 “어디까지 허용할 것인가”를 먼저 정해야 한다.

### 추천 사용 시나리오

- 반복 리서치, 코드 작업, 문서 작업, 자동 검토를 하나의 운영면으로 묶고 싶은 팀
- 로컬 실행과 cloud delegation을 함께 쓰는 환경
- 장기 메모리, automations, 브라우저/컴퓨터 사용까지 포함한 멀티 실행면이 필요한 조직

### 피해야 할 사용 시나리오

- 즉시성 높은 IDE 보조만 원하는 개인 사용자
- 운영 문서와 비용 구조를 읽고 관리할 시간적 여력이 없는 가벼운 실험

### 최근 업데이트 반영 관점

- Codex의 핵심은 이번 주에도 “새 데모 기능”보다 “플랜과 통제 항목을 어떻게 설명하느냐”에 있었다.
- 즉 경쟁력은 모델 쇼케이스보다 control plane 정합성, 권한 구조, usage transparency에 있다.

### 현재 시점 총평

Codex는 가장 넓은 운영면을 가진 도구다. 그만큼 강력하지만, 제대로 쓰려면 기능보다 먼저 정책과 경계를 읽어야 한다.

## 추천 사용자 유형

- 리포지토리 안에서 오래 일하고 셸 중심으로 검증까지 끝내고 싶다면 Claude Code가 가장 맞다.
- IDE 안에서 병렬 agent, 시각적 관리, 팀용 plugin 배포까지 보고 싶다면 Cursor가 가장 맞다.
- 로컬 작업, cloud delegation, automations, control plane을 한 덩어리로 묶고 싶다면 Codex가 가장 맞다.

## 이번 주 총평

이번 주 시장 신호는 명확했다. 승부처는 모델 데모가 아니라 운영면이다. Claude Code는 실행 감각과 복구성, Cursor는 팀 플랫폼화, Codex는 통제 가능한 다중 실행면에서 강점을 보였다. 반대로 세 도구 모두 권한 경계, 검토 책임, 비용 추적을 무시하면 장점이 곧 부담이 된다.
