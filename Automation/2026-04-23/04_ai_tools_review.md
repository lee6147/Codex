## 1. Claude Code

### 강점

- 2026-04-20과 2026-04-22 공식 릴리즈가 모두 병렬 작업, 세션 재개, MCP 연결, 플러그인 복구, 안정화에 집중한다.
- 긴 세션과 여러 에이전트를 실제로 굴리는 데 필요한 운영성 개선이 가장 빠르게 쌓이고 있다.
- 터미널 중심 워크플로와 고급 사용자가 원하는 직접 제어감이 여전히 강하다.

### 약점

- 공급 제약과 가격 정책 변동 가능성이 실제 제품 운영 리스크로 떠오르고 있다.
- GUI/데스크톱 면은 커지고 있지만, 여전히 툴링과 권한 설계를 사용자 쪽에서 깊게 이해해야 체감 성능이 나온다.
- 모델과 제품의 강점이 큰 만큼 사용량 관리 실패 시 비용과 정책 변화 영향을 크게 받을 수 있다.

### 추천 사용 시나리오

- 복잡한 코드베이스를 오래 붙잡고 refactor, 검증, 반복 수정까지 이어가야 하는 팀
- 터미널 중심의 숙련 개발자가 에이전트 병렬 실행과 세밀한 제어를 함께 원할 때
- MCP, 플러그인, 스킬, 세션 운영을 직접 설계할 수 있는 환경

### 피해야 할 사용 시나리오

- 최소 설정으로 바로 안정적인 기업 표준 경로가 필요한 조직
- 사용량 정책 변화에 민감한 예산 구조

### 최신 업데이트 반영 관찰

- `v2.1.117`은 “새 기능 과시”보다 병렬 에이전트와 메인 스레드 실행 안정화에 초점을 둔다.
- Claude Code는 지금 가장 강한 실행형 코딩 에이전트 중 하나지만, 그만큼 운영 난도와 비용 변동성도 함께 커지고 있다.

### 현재 시점 총평

Claude Code는 여전히 실행력 기준 최상위권이다. 다만 강점은 더 이상 모델 품질만이 아니라 “길고 복잡한 세션을 끝까지 끌고 가는 능력”에서 나온다.

## 2. Cursor

### 강점

- Cursor 3 이후 제품 정체성이 분명하다. IDE 보조가 아니라 에이전트 작업면 자체를 팔고 있다.
- Canvases, Agents Window, CLI Debug Mode 같은 요소가 시각 결과물과 병렬 작업에 강하다.
- 기업 자본, 보안 파트너십, 모델 공급 인프라까지 함께 묶이면서 플랫폼화 속도가 빠르다.

### 약점

- 이번 창에서는 새 공식 changelog가 없었다. 즉 제품 변화 메시지는 강하지만 공식 릴리즈 신호는 잠시 비어 있다.
- IDE 사용자가 기대하는 “정밀 수동 편집 흐름”과 agent-first 방향 사이의 긴장이 존재한다.
- 비용 구조와 모델 공급, 기업 전략 변화가 사용자 경험만큼 중요한 변수다.

### 추천 사용 시나리오

- 병렬 에이전트 화면, 시각 산출물, 작업면 전환이 중요한 팀
- IDE 안팎을 넘나드는 agent-first 환경을 빠르게 도입하려는 조직
- 제품 실험과 기업 배포를 함께 보는 팀

### 피해야 할 사용 시나리오

- 전통적 IDE 흐름을 거의 그대로 유지하고 싶은 사용자
- CLI 중심으로 최소 면적만 원하는 사용자

### 최신 업데이트 반영 관찰

- 공식 changelog 공백에도 불구하고 외부 기사와 최근 공식 항목을 보면 Cursor는 “에이전트 운영 콘솔”로 방향을 확정했다.
- Cursor의 경쟁력은 코드 완성 품질만이 아니라 화면, 병렬성, 기업 파트너십, 보안 레이어를 함께 묶는 데 있다.

### 현재 시점 총평

Cursor는 제품 전략상 가장 공격적으로 넓어지고 있다. 잘 맞는 팀에는 가장 풍부한 작업면이 될 수 있지만, IDE 순수주의자에게는 오히려 부담이 될 수 있다.

## 3. OpenAI Codex

### 강점

- 2026-04-16 제품 발표 이후 Codex는 컴퓨터 사용, 브라우저, 메모리, 반복 작업, SSH devbox 등으로 작업면을 급격히 넓혔다.
- 2026-04-23 관찰 기준 Help Center와 2026-04-22 기준 rate card가 빠르게 갱신되며 접근 경로와 관리자 통제가 더 구체화됐다.
- Codex app, CLI, IDE, web, cloud를 하나의 운영 체계처럼 묶는 방향이 분명하다.

### 약점

- 기능 범위가 넓은 만큼 “정확히 어디까지 무엇을 지원하는가”를 문서로 따라가야 한다.
- 실무자는 모델 성능보다 경로, 요금, 관리자 통제, 플러그인 제어를 함께 이해해야 한다.
- 단순 코딩 보조만 원하는 사용자에게는 작업면이 과하게 넓어 보일 수 있다.

### 추천 사용 시나리오

- 로컬 실행과 클라우드 위임, 조직 통제, 반복 작업을 한 세트로 보고 싶은 팀
- 앱 중심의 다중 에이전트 운영과 automations까지 함께 활용하려는 조직
- 관리자 통제와 문서화가 중요한 기업 환경

### 피해야 할 사용 시나리오

- 아주 가벼운 코드 보조만 필요한 개인 사용자
- 제품 범위와 정책 변화를 추적할 여력이 없는 소규모 팀

### 최신 업데이트 반영 관찰

- Codex의 최근 변화는 모델 자체보다 운영 경로와 정책 정리에 가깝다.
- 이는 Codex가 실험용 툴보다 조직 배포형 작업면으로 이동 중이라는 뜻이다.

### 현재 시점 총평

Codex는 지금 가장 빠르게 “작업면 전체”를 넓히는 쪽이다. 강점은 단일 기능보다 통합 표면과 관리자 제어에서 나온다.

## 주요 트렌드 관찰

1. 컨텍스트 검색, observability, 스킬, 보안 검증이 이제 모델 성능만큼 중요해졌다.
2. 코딩 도구 경쟁은 IDE 기능 경쟁에서 운영층 경쟁으로 이동했다.
3. 가격과 공급 정책, 관리자 통제, 문서화가 실제 채택 장벽을 좌우하기 시작했다.
4. 반복 작업과 백그라운드 실행이 “부가 기능”이 아니라 핵심 제품 축이 되고 있다.
