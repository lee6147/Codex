## 1. Claude Code

### 강점

- `v2.1.126`은 project purge, pasted auth code 로그인, PowerShell 7 탐지, PowerShell 기본 셸 전환 등 실제 세션 운영에서 자주 막히던 지점을 직접 겨냥했다.
- 릴리스 노트 흐름을 보면 Claude Code는 여전히 화려한 데모보다 긴 CLI 작업 루프를 덜 끊기게 만드는 데 강점이 있다.
- 셸 중심 작업, 긴 세션, 반복 수정 루프에서 체감되는 운영성 개선이 빠르다.

### 약점

- 이번 창의 변화는 대형 기능 발표보다 안정화 패치라서 외부에서 느끼는 모멘텀은 Cursor보다 약하게 보일 수 있다.
- 최근 읽을 만한 기사 대부분이 승인 UX, 프롬프트 주입, 자격 증명 범위를 먼저 경고한다. 권한 설계가 약하면 성능 장점이 그대로 리스크가 된다.
- 팀 차원에서 장기 자동화를 열기 전에 환경 분리와 승인 규칙을 먼저 잡아야 한다.

### 추천 사용 시나리오

- 한 세션 안에서 분석, 수정, 검증을 오래 이어 가는 CLI 중심 개발 작업
- 로컬 스크립트, 셸 명령, 리포지토리 탐색을 끊기지 않게 붙여 쓰는 자동화 흐름
- GUI보다 실행 일관성과 운영 복구성을 더 중시하는 사용자

### 피해야 할 사용 시나리오

- 승인 단계와 권한 경계를 설계하지 않은 채 곧바로 고권한 셸 자동화를 크게 여는 환경
- 시각적 관리와 팀용 통합 배포면이 우선인 조직

### 최근 업데이트 반영 관찰

- Claude Code의 현재 메시지는 “더 화려함”보다 “더 안 끊김”이다.
- 따라서 다음 비교 포인트는 기능 수보다 세션 안정성, 로그인 회복력, Windows 셸 친화성, 승인 UX가 되어야 한다.

### 현재 시점 총평

Claude Code는 여전히 가장 실전적인 CLI형 도구다. 이번 창에서는 새 기능 과시보다 장기 작업을 덜 끊기게 만드는 운영 품질을 보여줬다.

## 2. Cursor

### 강점

- 2026-04-30 Security Review와 2026-05-01 Team Marketplace의 조합은 이번 창에서 가장 강한 공식 서사를 만들었다.
- SDK, Security Review, Marketplace 흐름을 함께 보면 Cursor는 이제 “좋은 AI 에디터”보다 “운영 가능한 agent platform”으로 포지션을 넓히고 있다.
- 시각적 UX, 팀 배포, 사후 감사, 정책화된 확장을 한 제품에 묶으려는 방향이 분명하다.

### 약점

- 실행면이 넓어질수록 승인 UX와 권한 설계가 약할 때 사고 비용도 같이 커진다.
- 최근 보도된 PocketOS 사고처럼, 잘못된 환경 추정과 과도한 권한이 겹치면 회복 비용이 매우 커질 수 있다.
- 순수 CLI 중심 문화에서는 Cursor의 장점 일부가 덜 직접적으로 느껴질 수 있다.

### 추천 사용 시나리오

- IDE 안에서 병렬 작업과 작업 분리를 시각적으로 관리하려는 팀
- 팀 규칙 배포와 플러그인/스킬 유통까지 같이 다루려는 조직
- GUI와 에이전트 자동화를 함께 쓰고 싶은 사용자

### 피해야 할 사용 시나리오

- 최소한의 권한 체계만 가진 상태에서 대규모 agent 실행을 빠르게 여는 경우
- 편집기 플랫폼성보다 셸 직접성과 로컬 단순함을 더 중시하는 환경

### 최근 업데이트 반영 관찰

- Cursor는 이번 주에 보안 검토와 팀 배포를 전면에 올리며 “확장 가능한 조직용 제품” 이미지를 굳혔다.
- 공식 메시지의 무게중심이 단순 코드 생성에서 운영과 감사로 이동했다는 점이 중요하다.

### 현재 시점 총평

Cursor는 이번 창에서 가장 명확한 확장 모멘텀을 보여 줬다. 다만 그 확장성은 권한 통제와 감사 체계를 감당할 준비가 된 팀에게 더 잘 맞는다.

## 3. OpenAI Codex

### 강점

- CLI `0.128.0`은 persisted `/goal`, 권한 프로필, 플러그인 흐름, 외부 세션 import 같은 기능으로 장기 작업 운영성을 강화했다.
- 플랜 문서와 보안 문서는 Memories, Automations, in-app browser, Computer Use, Codex Security를 한 운영 체계 안에서 읽히게 만든다.
- 조직 도입 시 비용, 권한, 데이터 제어, 보안 검증을 한 번에 설명할 수 있는 문서층이 상대적으로 탄탄하다.

### 약점

- 이번 창의 변화가 제품 데모보다 CLI 운영면과 문서 갱신 중심이라 외부에서 느끼는 즉각적인 모멘텀은 Cursor보다 약하다.
- Help Center 문서 일부는 절대 수정일을 숨겨 최신성 추적이 한 단계 불편하다.
- 초보 사용자는 기능보다 플랜과 권한 구조를 먼저 이해해야 해서 진입 장벽이 있다.

### 추천 사용 시나리오

- 자동화, 반복 리서치, 역할 분리, 권한 통제가 중요한 조직형 작업
- 코드 외에도 문서, 브라우저, 컴퓨터 사용까지 하나의 하네스로 묶고 싶은 팀
- 비용과 운영 정책을 명시적으로 관리할 수 있는 환경

### 피해야 할 사용 시나리오

- 즉시성 높은 IDE 보조만 원하는 개인 사용자
- 운영 문서와 권한 구조를 읽고 관리할 여력이 없는 작은 실험성 사용

### 최근 업데이트 반영 관찰

- 이번 창의 Codex 핵심은 “무엇을 더 보여 주나”보다 “어떻게 계속 굴리고 관리하나”였다.
- 즉 Codex의 경쟁력은 점점 모델 자체보다 control plane 품질과 운영 문서의 정합성으로 이동하고 있다.

### 현재 시점 총평

Codex는 가장 하네스 지향적이고 조직형 운영에 맞는 도구다. 이번 창의 문서·CLI 중심 흐름은 화려함은 약하지만 실제 도입 관점에서는 핵심적인 진전이다.
