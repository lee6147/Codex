## 도구별 평가

## 1. Claude Code

### 강점

- CLI와 저장소 중심 작업에 잘 맞는다.
- 로컬 파일 수정, 명령 실행, 긴 맥락 기반 작업을 하나의 세션에서 이어가기 좋다.
- 최근 릴리스 흐름은 플러그인 설치와 안정성 개선 쪽으로 보이며, 워크플로 확장 가능성이 크다.

### 약점

- 강한 권한을 주면 파일 변경 범위가 커질 수 있어 Git 상태와 테스트 체계가 약한 팀에는 위험하다.
- 공식 릴리스 노트의 세부 설명이 압축적이어서, 버전별 변화의 실무 영향은 직접 확인해야 한다.

### 추천 사용 시나리오

- CLI에 익숙한 개발자가 저장소 전체를 읽고 작은 패치, 리팩터링, 테스트 실행까지 이어가는 작업.
- 로컬 `AGENTS.md`, 스킬, 작업 계약을 갖춘 팀의 반복 자동화.

### 피해야 하는 사용 시나리오

- Git 백업이나 테스트 없이 큰 파일 트리를 맡기는 작업.
- 민감 파일 접근권한이 정리되지 않은 프로젝트.

### 현재 시점 총평

Claude Code는 깊은 로컬 작업에 강하지만, 강점이 곧 위험이 될 수 있다. 운영 표준은 "권한 축소, 변경 전 상태 확인, 변경 후 테스트"를 기본으로 잡아야 한다.

## 2. Cursor

### 강점

- IDE 안에서 코드 읽기, 편집, 에이전트 실행을 자연스럽게 연결한다.
- 2026-05-06 Cursor 3.3의 컨텍스트 사용량 표시와 에이전트 터미널 관리는 대형 프로젝트 작업 가시성을 높인다.
- 초중급 개발자도 CLI 도구보다 낮은 진입 장벽으로 AI 코딩 흐름을 경험할 수 있다.

### 약점

- 에이전트가 파일을 직접 바꾸는 경우에는 여전히 변경 범위 통제가 필요하다.
- 병렬 터미널과 에이전트 실행이 늘수록 실패 원인 추적이 복잡해질 수 있다.

### 추천 사용 시나리오

- IDE 중심으로 빠르게 기능을 수정하고, 사람이 화면에서 변경 내용을 계속 확인하는 작업.
- 컨텍스트 사용량을 보며 긴 세션을 관리해야 하는 대형 저장소 작업.

### 피해야 하는 사용 시나리오

- Git 상태를 확인하지 않고 에이전트에 광범위한 삭제·이동 작업을 맡기는 경우.
- 자동 실행 로그를 남기지 않는 병렬 작업.

### 현재 시점 총평

Cursor는 "보이는 에이전트 운영 UI" 쪽으로 진화하고 있다. 생산성은 높지만, 실제 팀 도입에서는 권한·로그·리뷰 정책이 같이 가야 한다.

## 3. OpenAI Codex

### 강점

- CLI 기반 자동 패치, 테스트 실행, 검증 보고에 맞는 구조를 갖는다.
- GitHub 릴리스에서 0.129.x alpha 흐름이 빠르게 확인되어 기능 변화 속도가 높다.
- 로컬 하네스와 결합하면 리포트 생성, 검증 스크립트, 파일 단위 산출물 관리에 강하다.

### 약점

- alpha 채널은 운영 안정성 기준으로 보기 어렵다.
- 도구 호출, 샌드박스, 승인 정책이 결과 품질에 직접 영향을 주므로 환경 설정을 이해해야 한다.

### 추천 사용 시나리오

- "파일 생성 또는 수정 -> 검증 스크립트 실행 -> 근거와 결과 보고"가 필요한 자동화 작업.
- 개발자가 최종 검토하고 병합하는 보조 에이전트 역할.

### 피해야 하는 사용 시나리오

- 외부 운영 시스템에 직접 쓰는 작업.
- 테스트나 검증 스크립트가 없는 상태에서 넓은 범위의 리팩터링을 맡기는 작업.

### 현재 시점 총평

Codex는 검증 가능한 자동화 작업에 강하다. 오늘 보고서 작업처럼 로컬 계약과 종료 검증이 있는 업무에서는 특히 잘 맞지만, alpha 릴리스는 별도 실험 경로로 분리해야 한다.

1. 에이전트 실행 권한 관리가 제품 기능만큼 중요해졌다.
