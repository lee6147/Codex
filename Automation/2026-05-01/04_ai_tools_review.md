## 1. Claude Code

### 강점

- 2026-04-28과 2026-04-29에 걸친 `v2.1.121`~`v2.1.123` 연속 릴리스는 인증, ToolSearch, PR URL 검색, Bedrock 리전 선택처럼 실제 작업 루프에서 자주 걸리는 지점을 빠르게 보정했다.
- 공식 Week 17 문맥까지 함께 보면 Claude Code는 여전히 긴 CLI 작업, 다회 수정, 세션 유지, 로컬 도구 연결에서 가장 단단한 인상을 준다.
- 새 기능을 크게 홍보하기보다 운영 안정성과 복구성을 밀도 있게 다듬는 흐름이 명확하다.

### 약점

- 이번 창의 변화는 대부분 운영 패치라서, 외부에서 보기에는 “새로운 그림”보다 “기존 사용자 경험 정리”에 가깝다.
- 기업 계정, OAuth, MCP, ToolSearch 같은 운영층 비중이 높아질수록 초기 설정 부담은 계속 남는다.
- 최근 외부 보안 기사 흐름까지 고려하면 권한 범위와 토큰 관리 정책이 더 민감해진다.

### 추천 사용 시나리오

- 긴 저장소에서 분석, 수정, 검증을 한 세션 안에서 끈기 있게 이어 가야 하는 작업
- MCP, hooks, 로컬 스크립트, 다회 셸 실행을 묶어 실제 작업 흐름을 만드는 팀
- IDE보다 CLI와 자동화 하네스를 중심으로 에이전트를 붙이는 조직

### 피해야 할 사용 시나리오

- 별도 운영층 설정 없이 가벼운 에디터 보조만 원하는 사용자
- 즉시 시각적 피드백과 GUI 중심 관리가 더 중요한 환경

### 최근 업데이트 반영 관찰

- Claude Code의 이번 주 메시지는 “더 화려해짐”이 아니라 “더 안 끊김”이다.
- 따라서 다음 비교 축은 기능 수보다 안정성, 복구성, 권한 연결 난이도가 되어야 한다.

### 현재 시점 총평

Claude Code는 여전히 가장 강한 CLI형 실전 도구다. 이번 창에서는 공격적 확장보다 기반 정비가 중심이었고, 그 방향은 오히려 장기 사용자에게 더 중요하다.

## 2. Cursor

### 강점

- 2026-04-29 Cursor SDK 공개는 이번 창에서 가장 큰 단일 제품 업데이트다.
- 직전 Cursor 3.2의 multitask, worktree, multi-root 철학이 SDK로 이어지면서, Cursor는 UI 중심 경험과 프로그래밍 호출면을 동시에 가져가려는 그림을 분명히 했다.
- 시각적 병렬 작업과 프로그래밍 가능한 에이전트 실행면을 한 제품 안에서 연결하는 점이 강하다.

### 약점

- 기능 확장이 빠른 만큼 설정 복잡도와 운영 규칙의 중요성도 더 커진다.
- IDE 친화성은 강하지만, 순수 CLI 중심 팀에는 Cursor의 강점이 덜 직접적으로 체감될 수 있다.
- 최근 시장 담론이 “에이전트를 더 많이 맡기는 구조”로 움직이면서, 보안과 비용 통제 질문도 더 빨리 따라붙는다.

### 추천 사용 시나리오

- 여러 작업을 시각적으로 병렬 관리하면서 필요하면 SDK로 자동화까지 연결하려는 팀
- IDE 중심 개발 문화가 강하고, worktree 기반 분리 작업이 잦은 조직
- UI 경험과 에이전트 자동화를 한 제품 안에서 같이 쓰고 싶은 사용자

### 피해야 할 사용 시나리오

- 화면보다 셸을 우선하는 순수 CLI 문화
- 최소 설정과 단순 도구 구성을 더 선호하는 환경

### 최근 업데이트 반영 관찰

- Cursor는 이번 주 “좋은 AI 에디터”에서 “프로그래밍 가능한 에이전트 플랫폼”으로 한 단계 더 전진했다.
- SDK가 실제로 자리 잡으면 Cursor의 경쟁 상대는 단순 IDE 보조가 아니라 작업 오케스트레이션 도구 전체가 된다.

### 현재 시점 총평

Cursor는 이번 창에서 가장 명확한 확장 메시지를 냈다. 다만 확장 속도가 빠른 만큼 권한, 비용, 정책 레이어를 함께 설계할 준비가 된 팀에게 더 유리하다.

## 3. OpenAI Codex

### 강점

- 최근 공식 문서 흐름은 Codex를 app, CLI, IDE extension, web, cloud delegation까지 아우르는 운영 체계로 설명한다.
- Symphony 이후 Codex의 방향은 단일 세션 도구보다 orchestration control plane에 더 가깝다.
- 플랜과 rate card 문서가 정리되면서 조직형 도입 관점에서는 제품 구조가 더 읽기 쉬워졌다.

### 약점

- 이번 3일 창에서는 새 기능보다 문서 갱신이 중심이라, 외부 체감상 제품 모멘텀은 상대적으로 덜 선명하다.
- Help Center는 절대 수정 시각을 주지 않아 최신성 검증이 항상 한 단계 불편하다.
- 초보 사용자는 기능 이해보다 플랜, 권한, 과금 구조를 먼저 읽어야 해서 진입 장벽이 남는다.

### 추천 사용 시나리오

- 반복 보고, 일정 기반 작업, 다중 에이전트 오케스트레이션, 권한 통제가 중요한 조직
- 코드 외에도 문서, 리서치, 자동화까지 한 하네스에서 묶어 운용하려는 팀
- 과금 구조와 실행 권한을 명시적으로 설계할 수 있는 환경

### 피해야 할 사용 시나리오

- 즉시성 높은 IDE 보조만 원하는 개인 사용자
- 운영 문서와 권한 구조를 읽고 조정할 여력이 없는 작은 실험성 사용

### 최근 업데이트 반영 관찰

- 이번 창의 Codex 핵심은 “무엇을 더 해 주는가”보다 “어떤 플랜과 비용 구조 안에서 굴릴 것인가”였다.
- 즉 기능 데모보다 운영 가시성이 더 중요해진 시점이다.

### 현재 시점 총평

Codex는 가장 하네스 지향적인 도구다. 이번 창에서 문서 갱신이 중심이었던 점은 화려함은 약하지만, 조직형 운영이라는 핵심 방향은 오히려 더 선명하게 만든다.

## 추천 사용자 유형

- 한 세션에서 길게 파고드는 코드 작업과 수정 루프가 핵심이면 Claude Code가 가장 맞는다.

## 이번 주 총평

AI 코딩 도구 시장은 “어느 모델이 더 똑똑한가”보다 “에이전트를 얼마나 안정적으로, 많이, 구조적으로 굴릴 수 있는가”로 빠르게 이동하고 있다. Claude Code는 안정화, Cursor는 실행면 확장, Codex는 운영 체계 정리라는 서로 다른 방식으로 같은 방향을 향하고 있다.
