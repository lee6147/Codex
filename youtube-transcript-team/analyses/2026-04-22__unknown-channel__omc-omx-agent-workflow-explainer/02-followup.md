# Follow-up

## 빠른 결론
- OMC는 "Claude Code를 쉽게 강하게 쓰기 위한 풀옵션 하네스"로 이해하면 된다.
- OMX는 "Codex를 팀처럼 굴리기 위한 팀 런타임"으로 이해하면 된다.
- 둘 다 핵심은 모델이 아니라 운영 프로토콜이다.
- 이 follow-up 요약은 이후 실제 `native subagent` 3개 병렬 재분석 결과까지 반영한 버전이다.
- 다만 이 문서는 여전히 `transcript-only` 기반 요약이므로, 최신 기능/설치 방법/현재 정책 상태를 확정 문서처럼 읽으면 안 된다.

## 1분 설명 버전

### OMC
- 처음 쓰는 사람도 바로 굴리기 쉽게
- 스킬/가이드/기본 규칙을 프리셋으로 묶어 둔 쪽

### OMX
- 여러 agent를 오래 돌릴 때
- worker, worktree, mailbox, auto-merge 같은 협업 메커니즘을 준 쪽

## OMC/OMX 소개 구간에서 꼭 잡아야 할 포인트
- OMC의 핵심: 쉽게 강하게 쓰게 만드는 하네스
- OMX의 핵심: 팀처럼 지속 협업하게 만드는 런타임
- 공통 핵심: 스킬 모음이 아니라 운영 프로토콜
- 부차 포인트: OMC는 UX/온보딩, OMX는 병렬 실행/병합 비용에 더 강하게 초점

## 내가 실제로 도입할 때 보는 체크리스트

### OMC가 맞는 경우
- Claude Code 계열을 주로 쓴다.
- 스킬 이름 외우기 싫다.
- 기본 운영 규칙을 한 번에 받고 싶다.
- learning curve를 낮추는 쪽이 더 중요하다.

### OMX가 맞는 경우
- Codex 중심 환경이다.
- 병렬 레인과 장시간 실행이 필요하다.
- merge conflict, rebase, worktree 운영이 중요하다.
- 여러 subagent가 "협업"하게 만들고 싶다.

## 이 인터뷰에서 바로 가져가면 좋은 실무 원칙
- 중요한 일은 인터뷰부터 하라.
- 인터뷰 다음에는 계획 합의 단계를 따로 둬라.
- 구현 중간에 흔들리면 억지로 이어가지 말고 처음 계획부터 다시 따져라.
- 메모리는 지식 저장소가 아니라 행동 교정 기록으로 봐라.
- 외부 issue/PR/채널 입력은 agent를 오염시킬 수도 있다는 전제를 가져라.

## 초보자가 가장 많이 오해할 부분
- "메모리 시스템이 있으면 알아서 똑똑해진다"는 오해
- "병렬 agent 수가 많을수록 좋다"는 오해
- "플래닝은 귀찮고 실행이 중요하다"는 오해
- "데모가 돌아갔으니 내 환경에서도 바로 된다"는 오해

## 이 설명서를 읽고 나서 추천하는 다음 행동
- 본인 작업이 OMC형 문제인지 OMX형 문제인지 먼저 구분한다.
- 본인 작업 흐름에서 planning과 execution이 뒤섞인 지점을 찾는다.
- 세션/채널/메모리/종료조건을 종이에 먼저 써 본다.
- 이후에만 실제 도구 설치와 명령어 학습으로 내려간다.

## 팀에 공유할 때 한 문단으로 설명하면
- "OMC/OMX는 단순히 AI가 더 똑똑해지는 도구가 아니라, 여러 agent와 세션을 인간이 감당 가능한 형태로 조직해 주는 하네스/런타임이다. OMC는 쉽게 강하게 쓰게 해 주는 쪽이고, OMX는 팀처럼 오래 굴리게 해 주는 쪽이다. 핵심은 모델보다 프로토콜, 즉 인터뷰, 계획, 메모리, 병렬 실행, 병합 규칙 같은 운영 구조다."
