# YouTube Transcript Agent Team

이 폴더는 유튜브 영상 스크립트를 붙여 넣었을 때 빠르게 판단할 수 있도록 만든 로컬 에이전트 팀 설계안이다.

핵심 목적은 네 가지다.

1. 긴 영상을 직접 보지 않고도 핵심만 파악한다.
2. 정말 시간을 들여 풀영상을 봐야 하는지 구분한다.
3. 실제로 적용할 아이디어와 그냥 흘려들어도 되는 이야기를 분리한다.
4. 그럴듯하지만 검증되지 않은 주장에 속지 않는다.

## 이 팀이 잘 작동하려면

아래 네 가지가 들어오면 결과 품질이 확 올라간다.

- 영상 제목과 채널명
- 스크립트가 전체인지 일부인지
- 사용자의 현재 수준
- 이전에 본 비슷한 영상 목록

특히 `Companion Recommender`는 이전에 본 영상 목록이 있어야 훨씬 정확해진다.
없으면 특정 영상 추천 대신 "어떤 유형의 영상을 찾아보면 좋은지" 중심으로만 답한다.

## Git / Worktree 메모

- 이 워크스페이스가 Git 저장소라면 `git worktree`를 써서 병렬 실험이나 브랜치 분리를 할 수 있다.
- 다만 유튜브 분석 산출물은 기본적으로 현재 메인 워크스페이스의 `youtube-transcript-team/analyses/` 아래에 저장한다.
- 별도 worktree를 쓰더라도 분석 결과 보관 위치는 사용자가 따로 원하지 않는 한 이 폴더를 기준으로 유지한다.

## 추천 팀 구성

### 1) Summary Analyst
- 영상의 핵심 주장, 구조, 예시, 결론을 압축한다.
- "이 영상이 실제로 말하는 것"과 "말만 길고 실속 없는 부분"을 분리한다.

### 2) Watchworthiness Judge
- 이 영상이 진짜 시간을 투자해서 볼 가치가 있는지 판정한다.
- 요약만 봐도 충분한 영상인지, 직접 봐야 하는 영상인지 이유와 함께 말한다.

### 3) Novelty Detector
- 이미 많이 나온 이야기인지, 새로운 관점이나 실전 팁이 있는지 판정한다.
- 클로드코드, 코덱스, MCP, 멀티에이전트, 자동화 같은 주제에서 흔한 클리셰와 구분한다.

### 4) Action Curator
- 사용자가 당장 해볼 만한 액션만 뽑아낸다.
- 설치, 프롬프트 습관, 워크플로우 변경, 팀 적용 포인트를 실행 단위로 바꾼다.

### 5) Practicality Evaluator
- 영상의 내용이 실제 업무에 적용 가능한지 판정한다.
- 툴 제약, 유료 플랜, 팀 환경, 전제조건이 빠져 있지 않은지 본다.

### 6) Fact Checker (선택)
- 모델명, 기능 지원 여부, 날짜 민감 정보, 가격/정책처럼 빨리 바뀌는 요소를 따로 표시한다.
- 스크립트만으로 검증이 안 되는 항목은 사실처럼 단정하지 않고 "검증 필요"로 남긴다.

### 7) Skeptic Reviewer (선택)
- 과장, 검증 불가 주장, 판매성 문장, 플랫폼 락인 유도, 근거 없는 생산성 수치를 의심한다.
- 영상이 "정보"보다 "홍보"에 가까운지 판정한다.

### 8) Companion Recommender (선택)
- 이 영상과 같이 보면 이해가 더 잘 되는 이전 영상 유형을 추천한다.
- "먼저 볼 영상", "같이 볼 영상", "굳이 안 이어봐도 되는 영상"을 구분한다.

## 권장 운영 순서

1. Summary Analyst가 내용을 압축한다.
2. 기본 모드에서는 Summary Analyst가 신규성까지 가볍게 같이 본다.
3. 이미 비슷한 영상을 많이 본 상태이거나 "또 같은 얘기인지"가 중요하면 Novelty Detector를 별도로 붙인다.
4. Practicality Evaluator가 실제 적용 가능성을 판정한다.
5. Watchworthiness Judge가 시청 가치와 시간 투자 여부를 판정한다.
6. Action Curator가 실천 항목을 정리한다.
7. Fact Checker와 Skeptic Reviewer는 긴 영상, 판매성 강한 영상, 최신 정보 의존 영상에서만 추가로 돌린다.
8. 이미 본 영상 목록이나 채널 맥락이 있으면 Companion Recommender를 붙여서 연속 시청 순서를 추천한다.

## 최종 산출물 형식

매번 아래 형식으로 끝내면 좋다.

1. 세 줄 요약
2. 핵심 주장 3~5개
3. 진짜 새로운 내용
4. 뻔한 내용 / 과장된 내용
5. 실행 가능성: `높음 / 조건부 / 낮음`
6. 지금 직접 볼 가치: `높음 / 보통 / 낮음`
7. 직접 봐야 하는 이유 또는 안 봐도 되는 이유
8. 내가 바로 해볼 액션 3개
9. 검증이 필요한 주장
10. 데모/증거 유무
11. 추천 대상: `초보 / 실무자 / 팀 리드`
12. 같이 보면 좋은 이전 영상 유형
13. 판단 신뢰도: `높음 / 보통 / 낮음`

## 빠른 사용법

`paste-template.md`의 형식으로 영상 정보와 스크립트를 붙여 넣고, `prompts/orchestrator.md`를 리더 프롬프트로 쓰면 된다.
이 풀 버전 분석이 기본 모드다.

가볍게 쓰고 싶으면 `quick-template.md`와 `quick-mode.md`를 쓴다.
이 모드는 긴 구조 대신 핵심 판단만 먼저 뽑아주는 1차 판정용이다.
사용자가 `짧게`, `압축해서`, `결론만`처럼 요청할 때 이 모드로 전환한다.

짧은 영상이나 정보 밀도가 낮은 영상은 3인 체제로도 충분하다.
- Summary Analyst
- Watchworthiness Judge
- Action Curator

긴 영상, 비교 영상, "2026 트렌드", "AI로 10배 생산성" 같은 제목은 5인 체제가 더 안전하다.

## 가장 실용적인 추천

바쁜 사용자를 기준으로 한 기본 조합은 4인 팀이다.

- Summary Analyst
- Watchworthiness Judge
- Practicality Evaluator
- Action Curator

여기에 아래 조건이면 보조 역할을 붙인다.

- 최신 기능, 가격, 정책이 핵심이면 `Fact Checker`
- 영상이 유난히 자극적이거나 판매성이 강하면 `Skeptic Reviewer`
- 이미 비슷한 영상을 많이 본 상태이거나 중복 제거가 중요하면 `Novelty Detector`
- 이미 본 영상 목록이 있거나 시리즈형 학습을 하고 싶으면 `Companion Recommender`

## 운영 팁

- 스크립트가 일부만 있거나 자동 자막 품질이 나쁘면 판단 신뢰도를 낮춰야 한다.
- 데모 중심 영상은 요약 정확도보다 `직접 시청 필요 여부` 판정이 더 중요하다.
- 최신 모델, 가격, 정책 비교는 transcript-only 환경에서 오답이 나기 쉬우므로 `Fact Checker`를 붙여도 확정 표현을 피해야 한다.
- 고급 비교가 필요하지 않으면 `Novelty Detector`는 Summary 단계에 흡수해서 간단히 처리해도 된다.

모델별 흔들림을 줄이고 싶으면 [golden-examples.md](C:\Users\user\Desktop\Codex\youtube-transcript-team\golden-examples.md)를 기준 출력 예시로 같이 본다.

## 압축 모드

빨리 보고 결정만 하고 싶을 때는 아래 6개만 출력하면 된다.

1. 한줄 결론
2. 핵심 요약 3줄
3. 직접 볼 가치: `높음 / 보통 / 낮음`
4. 바로 실행할 것 3개
5. 검증 필요 주장
6. 같이 보면 좋은 영상 유형

자세한 분석이 필요할 때만 풀 버전으로 다시 돌린다.

## 분석 저장 방식

현재부터는 영상 하나를 분석할 때마다 `analyses/` 아래에 새 폴더를 만들어 저장한다.

권장 폴더 이름 형식:

- `YYYY-MM-DD__channel__video-slug`

예:

- `2026-04-15__anthropic__claude-code-tips`
- `2026-04-15__openai__codex-workflow-demo`

각 폴더에는 아래 3개 파일을 기본으로 둔다.

- `00-intake.md`
- 사용자가 준 영상 정보, 질문, 시청 이력, 스크립트 원문 또는 일부
- `01-analysis.md`
- 실제 분석 결과
- `02-followup.md`
- 바로 실행할 것, 다음에 볼 영상, 이후 비교 메모

짧게 분석했으면 `01-analysis.md`에 압축 모드 결과를 쓰고, 자세히 분석했으면 풀 버전 결과를 쓴다.

원문 스크립트가 짧으면 `00-intake.md` 안에 직접 저장한다.
원문이 너무 길면 같은 영상 폴더 안에 `transcript/` 하위 폴더를 만들고 아래처럼 나눠 저장한다.

- `transcript/transcript-part-01.md`
- `transcript/transcript-part-02.md`
- `transcript/transcript-part-03.md`

이 경우 `00-intake.md`에는 전체 메타 정보와 함께 `원문은 transcript/ 하위 파일 참조`라고 적고, 누락 없이 파트 목록을 남긴다.
